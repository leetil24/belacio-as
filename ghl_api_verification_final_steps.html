Phase 1 Setup
1
Get your Private Integration Token
GHL sub-account → Settings → Integrations → Private Integration → copy token. This is your auth credential for every Postman call in this checklist.
2
Configure Postman environment
Base URL: https://services.leadconnectorhq.com. Default headers on every request: Authorization: Bearer {token} and Version: 2021-07-28. Save locationId as an environment variable so all calls share it.
Phase 2 Auth verification
3
Confirm token and scopes
GET /contacts/?locationId={id}&limit=1 → expect 200 OK with a contacts array. A 401 = bad token. A 403 = missing scope. Do not proceed past this step until you have a clean 200.
Screenshot: 200 OK + response body
Phase 3 Product structure
4
Confirm where WooCommerce order data lives in GHL added
Before pulling field IDs, confirm the data model: does order/product data map to Contact custom fields, Opportunity custom fields, or a Custom Object? Check with your GHL architect or inspect existing records. Getting this wrong means Franz builds field mappings against the wrong object.
Decision gate — confirm before Step 5
5
Retrieve all relevant custom fields and copy their IDs
GET /locations/{locationId}/customFields → identify every field that maps to WooCommerce data: order ID, product SKU, stock quantity, order total, fulfillment status. Copy each field's exact id value into a reference table — these get hardcoded into every n8n workflow.
Screenshot: full field list with IDs visible
Phase 4 Inventory update
6
Write a stock value to the correct record
PUT /contacts/{contactId} with body {"customFields":[{"id":"{fieldId}","field_value":50}]}. Confirm the numeric value appears in the 200 response and is visible in the GHL UI. If inventory data lives on an Opportunity instead, use PUT /opportunities/{id} with the same pattern.
Screenshot: 200 response + updated field value in GHL UI
Phase 5 Trigger events
7
Test contact upsert — no duplicate creation added
POST /contacts/upsert with a body containing an email that already exists. Confirm the response updates the existing contact rather than creating a new one. This is the endpoint WF 2 uses — if untested, you risk duplicate contacts in production every time a WooCommerce order comes in.
Screenshot: 200 response showing updated existing contact
8
Set up a test webhook receiver
In n8n, create a Webhook trigger node and copy its test URL. Alternatively use webhook.site. This is your listener for the outbound event test below.
9
Fire a GHL outbound event and inspect the payload
In GHL, manually trigger an event (e.g. move an opportunity stage). Confirm delivery at your receiver. Inspect exactly which fields arrive — GHL payloads do not include full contact data, so document what supplementary GET /contacts/{id} calls n8n will need to make on every trigger.
Screenshot: payload received showing all fields present
10
Test GHL inbound webhook trigger added
In GHL, open the workflow that WooCommerce/n8n will call. Copy its inbound webhook URL. From Postman, send a POST to that URL with a sample WooCommerce order payload. Confirm GHL returns 200 and the workflow execution log shows it fired. This validates the receive path for WF 2 and WF 3.
Screenshot: Postman 200 + GHL workflow execution log showing trigger fired
Phase 6 Pipeline verification
11
Get the Retail Pipeline ID and all stage IDs
GET /opportunities/pipelines?locationId={id} → find the Retail Pipeline. Copy its pipelineId and every stage's id and name into a reference table. Franz hardcodes these into WF 1, 3, and 4 — they cannot be looked up dynamically at runtime.
Screenshot: pipeline object with full stages array visible
12
Create a test opportunity card
POST /opportunities with pipelineId, stageId, contactId, and a test name. Confirm the card appears on the Retail Pipeline board in the GHL UI. Then delete the test card.
Screenshot: 200 response + card visible on pipeline board
13
Move the opportunity to a different stage
PUT /opportunities/{id} with a different stageId. Confirm the card moves to the new stage in the GHL UI. This validates the full write path every n8n workflow uses when updating order status.
Screenshot: 200 response + card in updated stage
Phase 7 Deliverable
14
Build the reference tables
From your notes compile two tables: (1) Custom field ID reference — field name, object type, field ID, data type. (2) Pipeline stage ID reference — stage name, stage ID. These ship alongside the screenshots as the primary deliverable Franz needs.
15
Compile screenshots into the five acceptance criteria
Organize all screenshots under: auth, product structure, inventory update, trigger events, pipeline. Each section should show the endpoint, request body, and 200 response. Attach both reference tables.
